# Policy-Driven Hexagonal Scaffolding Framework — 설계안 (v0.5)

> 정책·문서 요구사항을 입력하면, 헥사고날·클린 아키텍처를 준수하는 프로젝트 골격을 생성하고,
> 정책을 코어에 강제로 박으며, 변경 전파를 빌드 단계에서 차단하는 메타 프레임워크.

> **이 문서를 받는 AI 세션에게:** 아래 예시는 *형태를 합의하기 위한 것*이다. "헥사고날을 준수하라"는
> 추상 지시는 세션마다 다르게 해석되므로, **§3의 골격 예시와 정확히 같은 구조**로 산출하라.
> 예시는 가볍지만 구조(레이어·의존성 방향·파일 배치)는 그대로 따른다. 도메인 이름·필드는 바꿔도 된다.

---

## 📌 제1목표 (이 문서 전체를 지배하는 단일 원칙)

이 프레임워크가 내뱉는 **결과물(생성된 프로젝트)** 은 다음을 동시에 만족해야 한다.

1. **헥사고날 · 클린 아키텍처를 준수** — 모든 판단(정책·상태·불변식)은 도메인 코어에, 입출력은 어댑터에.
2. **정책·요구사항이 아무리 바뀌어도 견딘다** — 변경이 들어오면 충돌·모순이 코드 레벨에서 즉시 드러난다.
3. **지속적 바이브코딩에 견딘다** — AI가 코드를 계속 쌓고 바꿔도 경계가 무너지지 않는다.
4. **의도치 않은 버그·충돌을 방지한다** — 말단 요구가 뿌리(상태 모델·불변식)를 흔들면 빌드가 막는다.

> ⚠️ 이 4가지를 만족하지 못하는 산출물은 실패로 간주한다.

설계 철학 한 줄: **"자동 생성"이 아니라 "명시적 모델 + 자동 강제"**.

## 1. 🔴 핵심 메커니즘 — 변경 전파 차단 (CRITICAL)

문제의 본질은 **변경 전파**다. "임시저장 버튼 추가" 같은 말단 요구가 사실은 상태 모델(정책)을 건드리고,
불변식을 깨서, 데이터가 비고, 땜질하다 버그가 번진다. 이건 구조(계층화)가 아니라 *변경이 번지는 방식*의 문제다.

해법은 세 방어선이며 **전부 도메인 코어에 박힌다. 어댑터가 아니다.**

- 🔴 **방어선 A — 상태 전이를 Aggregate에 박기.** 허용 안 된 전이는 코어가 거부. 어댑터는 호출만.
- 🔴 **방어선 B — 불변식을 생성자·전이에 강제.** 불변식 깨는 상태로는 객체 생성조차 불가.
- 🔴 **방어선 C — 경계 위반 CI 가드.** import 규칙·판단 누출을 빌드가 막음.

세 방어선의 실제 코드 형태는 §3에서 예시로 고정한다.

---

## 2. 🔴 판단/입출력 분리 규칙 (CRITICAL — 세션 간 해석 차이를 없애는 핵심)

"헥사고날 준수"가 갈리는 1순위 지점이 이거다. **단 하나의 규칙으로 고정한다:**

> **판단은 코어, 입출력은 어댑터.**
> - "검수를 통과시킬지 *결정*" = 코어 (도메인/정책)
> - "결과를 DB에 *쓰기* / Kafka에 *발행*" = 어댑터

판별 체크리스트 (생성 시·바이브코딩 시 항상 적용):

- `if 상태/정책/권한` 분기가 컨트롤러·serializer·view에 있으면 → **위반.** 코어로 옮긴다.
- 도메인 파일이 ORM·HTTP·SDK를 import하면 → **위반.** 포트 뒤로 숨긴다.
- 도메인 메서드가 외부 시스템을 직접 호출하면 → **위반.** 아웃바운드 포트로 바꾼다.

## 3. 🔴 표준 골격 — 이 구조를 그대로 산출하라 (CRITICAL)

> **이것이 "헥사고날 준수"의 조작적 정의다.** 세션이 무엇을 만들든 디렉토리·의존성 방향은 아래와 동일.

### 3.1 디렉토리 트리 (도메인 1개 기준, 가벼운 예시)

```
src/
  contexts/
    orders/                      # 바운디드 컨텍스트(=모듈). 큰 개념 단위
      domain/                    # 순수 코어. 프레임워크 import 금지
        order.py                 # Aggregate: 상태 + 불변식 (방어선 A·B)
        order_state.py           # 상태 enum + 전이 테이블
      application/               # 유스케이스 + 인바운드 포트
        place_order.py
        ports_in.py
      ports/                     # 아웃바운드 포트 (인터페이스만)
        order_repository.py
        event_publisher.py
      policies/                  # 정책 객체 + 결정표
        order_policies.py
        decision_tables/
          discount.csv
      adapters/                  # 입출력만. 판단 금지
        rest/order_controller.py
        db/order_repository_pg.py
        messaging/kafka_publisher.py
      requirements/              # 가변 요구사항 (§5)
        REQUIREMENTS.md
  shared/
    events.py                    # 도메인 간 이벤트 계약
.arch/
  import-rules.toml              # 방어선 C 설정
  invariants/                    # 불변식 테스트 모음
```

의존성 방향(클린 아키텍처 동심원): `adapters → application → domain`, 그리고 `adapters → ports(인터페이스) ← application`.
**domain은 누구도 import하지 않는다(가장 안쪽).**

### 3.2 방어선 A — 상태 전이 (언어 무관 의사코드, 형태만 고정)

```python
# domain/order_state.py
class OrderState(Enum):
    DRAFT, SUBMITTED, APPROVED, REJECTED

# 허용 전이를 데이터로 명시. 여기 없는 전이는 거부된다.
ALLOWED = {
    DRAFT:     {SUBMITTED},
    SUBMITTED: {APPROVED, REJECTED},
}
```

```python
# domain/order.py — Aggregate
class Order:
    def submit(self):
        self._transition(OrderState.SUBMITTED)   # 어댑터는 이 메서드만 호출

    def _transition(self, to):
        if to not in ALLOWED[self.state]:
            raise IllegalTransition(self.state, to)   # 코어가 거부
        self.state = to
        # 구현 노트: 전이 후 도메인 이벤트를 여기서 *기록*만 한다(발행 X).
        #   실제 발행은 어댑터가 한다(§2). 트랜잭션 경계·이벤트 발행 시점은
        #   outbox 패턴으로 분리할 것 — at-least-once 중복 대비(§16.6).
```

> 새 상태(예: 임시저장) 추가 요구가 오면, **반드시 이 enum·테이블을 건드려야** 하므로
> "운영 반영 경로에 그 상태가 없다"는 충돌이 코어에서 즉시 드러난다.

### 3.3 방어선 B — 불변식

```python
# domain/order.py
def __init__(self, ...):
    if self.state == OrderState.APPROVED and self.payload is None:
        raise InvariantViolation("APPROVED는 payload 필수")  # 깨진 상태로 생성 불가
```

### 3.4 방어선 C — 경계 가드 (설정 형태 고정)

```toml
# .arch/import-rules.toml  (Python=import-linter / JVM=ArchUnit / Go=go-arch-lint 로 매핑)
[[rule]] name = "domain은 adapters 의존 금지"
  forbidden = "contexts.*.domain -> contexts.*.adapters"
[[rule]] name = "도메인 간 직접 의존 금지 (이벤트로만)"
  forbidden = "contexts.orders -> contexts.billing"
```

### 3.5 정책 결정표 (가변 분기를 표로 모음 — §2.2의 실체)

```csv
# policies/decision_tables/discount.csv
customer_type, order_total_gte, discount
A,            100000,          0.10
A,            0,               0.05
B,            0,               0.00
```

> "A형이면 이렇고 B형이면…" 같은 분기는 코드 if가 아니라 이 표 한 장에 모은다.
> 빈 칸·중복 행이 곧 모호함·모순의 시각화다.

## 4. 🟠 생성 엔진 — 스펙 → 골격 (HIGH, 결정론적·LLM 금지)

스펙(`domain-spec.yaml`)에서 §3 구조를 **결정론적으로** 생성. 구조가 흔들리면 제1목표가 무너지므로 LLM 미사용.
비즈니스 로직(포트 안쪽)만 🟡 AI가 채운다 — 시그니처·경계는 이미 고정.

### 4.1 스펙 IR 샘플 (가벼운 예시)

```yaml
context: orders
domains:
  Order:
    states: [DRAFT, SUBMITTED, APPROVED, REJECTED]
    transitions:
      DRAFT: [SUBMITTED]
      SUBMITTED: [APPROVED, REJECTED]
    invariants:
      - "APPROVED면 payload 필수"
    ports_out: [OrderRepository, EventPublisher]
    policies: [discount]
relations:
  - from: Order
    to: Billing
    via: event           # 직접 호출 금지
    event: OrderApproved
infrastructure:
  database: postgres
  messaging: kafka
```

> 계층화는 2단계까지(`domains` → 하위 필드). 그 아래 분기는 `decision_table`(csv)로. UI 단발 동작까지 쪼개는 과분해 금지.

## 5. 🔴 가변 요구사항 관리 (CRITICAL — 이전 버전의 최대 공백, 신규)

> 제1목표 2·3의 실체. "정책·디테일 요구사항이 바뀌는 것"을 *어디에 담고, 어떻게 추적·버전·충돌 관리*하는가.

### 5.1 요구사항은 어디 사는가

- **구조적 요구사항**(상태·불변식·관계) → `domain-spec.yaml` (단일 소스, §4.1)
- **디테일/가변 요구사항**(분기·세부 규칙) → 두 곳에만:
  - 규칙성 분기 → `policies/decision_tables/*.csv` (§3.5)
  - 서술형 세부 → `contexts/<도메인>/requirements/REQUIREMENTS.md`
- **UI 단발 동작** → 문서로 빼지 않고 구현 코드에 둔다(과분해 금지).

### 5.2 요구사항 ID와 추적 (traceability)

모든 요구사항은 **ID**를 가진다. ID로 요구사항 ↔ 코드 ↔ 테스트를 잇는다.

```markdown
# REQUIREMENTS.md (가벼운 예시)
- [ORD-012] APPROVED 주문은 운영 DB에 payload와 함께 반영된다.
  - 종류: 수정형 · 상태: active · 도입: 2026-06
  - 연결: domain/order.py::_transition, .arch/invariants/test_ord_012.py
- [ORD-018] CP A형은 10% 할인. (→ decision_tables/discount.csv)
  - 종류: 추가형 · 상태: active
```

코드/테스트에는 ID 주석을 단다 (`# req: ORD-012`). 변경 시 ID로 영향 범위를 역추적한다.

### 5.3 🟠 가변성의 4종류 — 종류별로 다르게 처리 (HIGH)

세션이 이걸 구분 못 하면, 가장 위험한 "수정형"을 "추가형"처럼 다뤄 사고를 재현한다.

| 종류 | 처리 | 위험 |
|------|------|------|
| **추가형** (새 상태·새 분기) | 전이 테이블·결정표에 행 추가 | 낮음 |
| **수정형** (기존 규칙 변경) | 기존 불변식·테스트와 충돌 검사 필수 | 🔴 높음 |
| **폐기형** (안 쓰는 규칙) | `상태: deprecated` 마킹 후 정리. 죽은 코드 추적 | 중간 |
| **조건부 공존** (구버전 A·신버전 B) | 결정표에 버전 컬럼 추가로 표현 | 🔴 높음 |

### 5.4 변경 이력·충돌 기록

정책/요구사항 변경 시 **결정 로그**를 남긴다 (경량 ADR).

```markdown
# requirements/CHANGELOG.md
## 2026-06-18 [ORD-012 수정형]
- 변경: 임시저장(DRAFT) 상태 추가 요청
- 충돌: 운영 반영 경로는 APPROVED만 가정 → payload 없는 DRAFT가 새면 데이터 빔
- 결정: DRAFT는 운영 반영 대상에서 제외(전이 테이블에 DRAFT→APPROVED 불허로 명시)
- 영향: order_state.py, test_ord_012.py
```

> 이 로그가 있으면 "이 변경을 되돌리면 뭐가 깨지나"를 역추적할 수 있다. 임시저장 사고가
> 재발하지 않는 이유는, 변경이 충돌을 *기록하게* 만들고 전이 테이블이 *거부하게* 만들기 때문이다.

## 6. 🟡 재생성 vs 보존 경계 (MEDIUM — 신규, 바이브코딩 누적의 핵심)

스펙이 바뀌어 재생성할 때, **골격은 덮어쓰고 AI가 짠 로직은 보존**한다.

- 골격(디렉토리·포트 시그니처·상태 enum) = 스펙에서 매번 재생성 (덮어씀)
- 구현(포트 안쪽 로직) = 보존. 보호 구역 마커로 구분:

```python
def place_order(...):
    # >>> generated: do-not-edit
    self._validate_ports()
    # <<< generated
    # >>> impl: editable  (AI 바이브코딩은 여기만)
    ...
    # <<< impl
```

> 재생성기는 `impl:editable` 블록을 건드리지 않는다. 그래서 스펙이 바뀌어도 쌓인 로직이 날아가지 않는다.

## 7. 🟡 아키텍처 자동 선택 (MEDIUM, LLM 사용)

정책 문서·서비스 규모를 LLM이 판단해 언어·프레임워크·DB·레이어를 선택. **단 §3 골격과 방어선은 어떤 선택에서도 불변.**

| 시그널 | 작은 규모 | 중간 | 큰 규모 |
|--------|-----------|------|---------|
| 언어/FW | Python/FastAPI | Django/Spring | Kotlin·Go |
| DB | PostgreSQL | PG + Redis | PG샤딩 + Kafka |
| 도메인 간 | 직접 호출 허용 | 이벤트 | Kafka 필수 |
| 레이어 | 2층 | 3층 | + CQRS 선택 |

🔴 **선택 무관 고정**: 코어 프레임워크 비의존 · 방어선 A·B·C · 정책 단일소스+결정표.
경계 가드는 언어별 도구로 자동 번역(Python `import-linter` / JVM `ArchUnit` / Go `go-arch-lint`).

## 8. 🟠 바이브코딩 프로토콜 (HIGH — 생성 프로젝트의 CLAUDE.md에 탑재)

1. 🔴 상태·정책 건드리는 변경은 **코드보다 영향 분석 먼저** ("어떤 상태·불변식·정책·요구사항ID를 건드리나").
2. 🟠 판단 로직은 코어에. 어댑터에 `if 정책` 금지 (§2).
3. 🟠 새 상태·전이는 반드시 전이 테이블 경유.
4. 🟠 변경 종류(추가/수정/폐기/공존) 먼저 분류 (§5.3).
5. 🟡 변경 후 불변식 테스트 전체 실행 + CHANGELOG 기록.

## 9. 🟠 검증 — "성공" 의 구체 기준 (HIGH — 신규)

세션이 무엇을 테스트해야 충분한지 명시. 생성 프로젝트는 최소 다음 테스트를 갖는다.

- ✅ **상태 전이 거부 테스트** — 허용 안 된 전이가 예외를 던지는가
- ✅ **불변식 테스트** — 불변식 깨는 객체가 생성 불가한가 (요구사항 ID별 1개)
- ✅ **경계 위반 테스트** — domain이 adapters를 import하면 CI 실패하는가
- ✅ **결정표 완전성** — 결정표에 빈 칸·모순 행이 없는가

이 4종이 전부 green이면 제1목표의 "버그·충돌 방지"가 기계적으로 보장된다.

## 10. 🔴 변경 게이트 — 거부권의 아키텍처 강제 (CRITICAL — 신규, 사람 의존성 제거)

> §5·§8은 "사람이 규율을 지킨다"는 전제에 기댄다. 이 섹션이 그 전제를 제거한다.
> 프레임워크는 변경을 *받아쓰는 도구*가 아니라 *거부권을 가진 게이트키퍼*다.
> 변경 요청이 들어오면 무조건 반영하지 않는다. 전체를 분석하고, 충돌하면 거부하거나 대안을 제시한다.
> 그리고 이 과정은 사람이 건너뛸 수 없다 — 우회로 자체가 없기 때문이다.

### 10.1 변경 처리 원칙 (제1원칙 → 게이트)

1. **의도 먼저 파악**: 변경 요청(정책·기획·디테일)이 *무엇을 달성하려는지* 깊게 분석한다. (받아쓰기 금지)
2. **전체 영향 분석**: 그 변경이 건드리는 상태·불변식·정책·요구사항 ID를 전부 탐색한다.
3. **충돌 판정**: 기존과 리졸브 가능한가를 **기계가 계산**해 통과/대안/거부로 분기한다.
4. **차단**: 충돌이 리졸브되기 전엔 다음 단계로 진행 불가. (조언이 아니라 차단)

### 10.2 🔴 장치 1 — 스펙이 머지의 유일한 관문 (the only door)

가장 근본적인 강제: **코드는 스펙을 거치지 않고는 바뀔 수 없다.**

- 상태·불변식·정책은 오직 `domain-spec.yaml`에서만 변경 가능. 코드 직접 수정은 §3.4 경계 가드가 빌드를 깬다.
- 따라서 모든 변경 요청은 **반드시 스펙 변경 PR**이 되고, 스펙 변경은 자동으로 게이트(§10.3)를 통과해야만 머지된다.
- "그냥 코드 고치면 되지"가 **물리적으로 불가능**해진다 — 이것이 사람 의존성을 없애는 1차 메커니즘.

### 10.3 🔴 장치 2 — 충돌 판정은 결정론적 기계가 계산 (사람 판단 아님)

"충돌하나?"를 사람이 눈으로 보지 않는다. 기계가 계산해 통과/거부를 정한다. 기계적으로 판정 가능한 충돌:

| 충돌 종류 | 판정 방법 | 결과 |
|-----------|-----------|------|
| 상태 전이 모순 | 전이 테이블에 없는 경로 요구 | 자동 거부 |
| 불변식 위반 | 변경 후 기존 불변식 테스트 깨짐 | 자동 거부 |
| 결정표 모순 | 같은 입력에 다른 출력 행 생성 | 자동 거부 |
| 요구사항 충돌 | 같은 ID active 요구사항과 모순 | 자동 거부 |

> §9 검증을 **변경 시점으로 앞당긴** 것이다. 사후 "어 깨졌네"가 아니라, 머지 전 게이트가 막는다.
> 사람의 "괜찮을 거야"가 끼어들 틈이 없다.

### 10.4 🟠 장치 3 — 거부 시 출구는 3개뿐

게이트가 거부하면 사람에게 주는 선택지를 3개로 제한한다. 아무거나 못 한다.

1. **철회(deto)** — 변경 포기. 가장 단순. *대안 생성이 너무 복잡하면 이것만 제공해도 된다.*
2. **대안 수용** — 게이트가 제시한 옵션 중 선택 (예: "DRAFT를 운영 반영 대상에서 제외하면 통과").
3. **명시적 override** — 강행하려면 **그 충돌을 해소하는 스펙 변경을 함께 제출**해야만 한다.
   즉 "기존 불변식을 X→Y로 바꾸겠다"를 명시 선언. 그러면 우회가 아니라 *기록된 결정*이 된다.

> override조차 "무시"가 아니라 "기존 규칙을 바꾸는 새 결정"으로만 가능하다.
> CHANGELOG(§5.4)에 자동으로 "ORD-012의 불변식을 X→Y로 변경"이 박힌다. 변경은 반드시 흔적을 남긴다.

### 10.5 🟠 장치 4 — 막는 건 결정론, 푸는 길은 LLM

역할 분담:

- **충돌 판정(거부 여부)** = 결정론적 기계. LLM이 "통과시켜도 될 듯요" 헛소리 못 하게.
- **대안 제시(해소 방법)** = LLM. "상태를 하나 더 쪼개거나, 운영 반영 조건을 바꾸면 풀린다" 같은 창의적 제안.

> 막는 건 기계가, 푸는 길은 AI가. **대안 생성이 버거울 만큼 복잡하면 대안을 포기하고 결정론적 거부만 남긴다.**
> 안전이 항상 우선이다.

### 10.6 왜 이것이 사람 의존성을 없애나

| 장치 | 무엇과 무관해지나 |
|------|-------------------|
| 1 (유일 관문) | 사람의 의지 — 우회로가 없어 안 지킬 수가 없음 |
| 2 (자동 판정) | 사람의 주의력 — 충돌을 놓칠 수가 없음 |
| 3 (3출구·override) | 사람의 망각 — 무시하려면 명시적 결정을 남겨야 함 |
| 4 (결정론 판정) | 사람의 컨디션 — 막는 기계엔 변덕이 없음 |

> 규율을 *지키게 만드는 게 아니라, 안 지키는 게 불가능하게* 만든다. 규율이 사람의 미덕이 아니라 시스템의 속성이 된다.

## 12. 처리 파이프라인

### 12.1 최초 생성

```
정책/문서
  │ 🟡[LLM] extract: 문서 → 스펙 IR (사람 검수)
  ▼
domain-spec.yaml  ◀── 단일 진실 공급원
  │ 🟡[LLM] select-arch: 규모 판단 → 스택 선택
  ├─ 🟠[결정론] scaffold: §3 골격 + 마이그레이션
  ├─ 🔴[코어] 상태 전이·불변식 (§3.2/3.3)
  ├─ 🟠[정책] 결정표 + 모순 검사 (§3.5)
  ├─ 🟡[AI] impl 블록 채움 (§6 보호구역)
  └─ 🔴[CI] check: 경계·불변식·완전성 (§9)
```

### 12.2 변경 흐름 (게이트 경유 — §10)

```
변경 요청 (정책·기획·디테일)
  │ 🟡[LLM] propose: 의도 파악 + 전체 영향 분석
  ▼
🔴[결정론] gate: 충돌 자동 판정
  ├─ 충돌 없음 ──────────→ 스펙 PR 머지 → 재생성(§6) → check(§9)
  ├─ 부분 충돌 → 🟡[LLM] 대안 제시 → 사람 선택 → 재판정
  └─ 리졸브 불가 ────────→ 거부(veto). 진행 차단. 출구는 철회/override만(§10.4)
```

> 변경은 코드 직접 수정이 아니라 **반드시 스펙 PR**로만 들어온다(§10.2). 게이트를 통과 못 하면 머지 불가.

## 16. 🔧 구현 노트 — 종이에서 코드로 넘어갈 때 부딪힐 것들 (구현 전 필독)

> 이 문서는 설계로서는 일관적이지만, 아직 **종이 위에서만 일관적**이다. 아래는 실제 구현하면
> 반드시 부딪히는 미해결·미검증 지점이다. 구현자는 이걸 "나중에 발견할 함정"이 아니라
> "처음부터 결정해야 할 항목"으로 다뤄야 한다. 각 항목에 권장 시작점을 달았다.

### 16.1 🔴 충돌 자동 판정의 실제 커버리지 (§10.3의 최대 미검증 지점)

- **문제**: "충돌을 기계가 자동 판정한다"는 핵심 주장인데, *어디까지* 진짜 기계로 되는지는 만들어봐야 안다.
  - ✅ 기계가 확실히 잡음: 전이 테이블에 없는 경로, 불변식 테스트 깨짐, 결정표 중복 행, 같은 ID 모순.
  - ⚠️ 기계가 못 잡음: "이 정책이 저 기획 의도와 충돌한다" 같은 **의미적 충돌**. 테스트로 표현 안 됨.
- **구현 결정 필요**: 의미적 충돌을 (a) LLM 보조 분석으로 *경고만* 띄울지, (b) 사람 리뷰 필수 단계로 둘지.
  - 권장: LLM은 "충돌 의심" 플래그만. **최종 차단은 결정론만** 신뢰(§10.5 원칙 유지). LLM 판정을 머지 게이트에 직접 연결하지 말 것 — 헛소리로 통과시킬 위험.
- **시작점**: 먼저 결정론 4종만 구현해 "자동 판정으로 막히는 변경 비율"을 측정. 그 비율이 낮으면 게이트의 실효성 자체를 재검토.

### 16.2 🔴 스펙-코드 드리프트의 실제 거동 (§6의 미검증 지점)

- **문제**: "골격은 재생성, 구현은 보호구역으로 보존"이 이론적으로만 정리됨. 바이브코딩이 보호구역
  **밖**(시그니처·상태 enum 등 generated 영역)을 건드리기 시작하면 재생성이 충돌한다.
- **구현 결정 필요**: 재생성 시 generated 영역과 현재 코드가 다르면 — (a) 강제 덮어쓰기, (b) 3-way merge,
  (c) 빌드 실패 후 사람 개입. 셋 중 무엇을 기본으로 할지.
  - 권장: 기본은 (c) 빌드 실패. "generated 영역이 수정됨"을 드리프트로 간주해 CI가 막는다. 보호구역 마커(§6)를 경계 가드(§3.4)와 같은 강제력으로 취급.
- **시작점**: Copier의 `--conflict` 동작을 실제 시나리오(스펙 변경 + impl 수정 동시)로 테스트. 마커가 깨지는 케이스를 먼저 재현.

### 16.3 🟠 결정표 "완전성·모순" 검사의 알고리즘 (§3.5 / §9의 미정 부분)

- **문제**: "빈 칸·모순 행이 없는가"를 검사한다고 했지만 *어떻게* 가 없다. 입력 조합이 많아지면 완전성 검사가 폭발한다.
- **구현 결정 필요**: 결정표를 단순 룩업으로 둘지, OPA/Rego나 의사결정 테이블 엔진(예: 표 → 규칙 컴파일)으로 둘지.
  - 권장: 작게는 csv + pandas로 "중복 키·미커버 조합" 검사부터. 규모 커지면 OPA로 이관.
- **주의**: 입력 도메인이 연속값(가격 등)이면 "모든 조합 커버"가 불가능 → 구간(range)으로 표현하고 구간 겹침/공백만 검사.

### 16.4 🟠 스펙 IR 스키마의 표현 한계 (§4.1)

- **문제**: 예시 YAML은 단순 케이스만 보여준다. 실제론 — 조건부 전이(특정 조건에서만 허용), 다중 Aggregate
  걸친 불변식, 시간 의존 규칙(유예기간) 등이 들어오면 현재 스키마로 표현이 안 된다.
- **구현 결정 필요**: 스키마를 어디까지 확장할지 vs 표현 못 하는 건 impl 코드로 내릴지의 경계.
  - 권장: 스키마는 **얇게** 유지(상태·전이·불변식·관계·정책 참조까지만). 그 이상 복잡한 규칙은 정책 객체(코드)로 내리되,
    그 정책도 §2(판단은 코어) 규칙을 따르게 한다. 스키마를 무한 확장하면 스펙이 곧 새 부채가 된다(§13).

### 16.5 🟠 LLM 단계(extract / select-arch / 대안제시)의 비결정성 (§3·§7·§10.5)

- **문제**: LLM 단계는 같은 입력에도 출력이 흔들린다. extract가 도메인을 매번 다르게 가르면 수렴성이 깨진다.
- **구현 결정 필요**: (a) LLM 출력을 사람이 확정하는 단계를 **반드시** 두기(이미 문서에 있음, 코드로 강제 필요),
  (b) 같은 스펙엔 같은 골격이 나오도록 골격 생성은 100% 결정론 유지(이미 원칙), (c) LLM 출력에 스키마 검증을 걸어 깨진 IR을 거부.
  - 권장: LLM 출력은 **항상 Pydantic 스키마 검증 통과** 후에만 다음 단계로. 검증 실패 시 재시도/사람 개입.

### 16.6 🟡 멀티 컨텍스트·이벤트 계약의 버전 관리 (§3.1 shared/events, §5.3 공존형)

- **문제**: 도메인 간 이벤트(`relations.via: event`)의 스키마가 바뀌면, 발행측·구독측이 동시에 안 바뀌어 깨진다.
  §5.3의 "조건부 공존"이 이벤트 레벨에서도 필요하다.
- **구현 결정 필요**: 이벤트 스키마 버저닝(v1/v2 공존), 호환성 검사를 게이트(§10)에 포함할지.
  - 권장: 이벤트 계약도 스펙의 일부로 관리하고, 하위호환 깨지는 변경은 게이트가 거부 대상으로 잡게 한다(추후).

### 16.7 🟡 "작은 규모는 강제 최소화"의 실제 분기 (§7 / §14)

- **문제**: 도메인 3개 내외엔 과설계라고 했는데, 생성기가 *어떻게* 강제를 줄일지 코드 분기가 없다.
- **구현 결정 필요**: 규모 티어에 따라 어떤 방어선을 끄고/켜는지 명시. (예: 작은 규모는 도메인 격리 import 규칙 생략, 이벤트 대신 직접 호출 허용)
  - 권장: 방어선 A·B·§9 검증은 **규모 무관 항상 ON**. 도메인 격리(C의 일부)·이벤트 강제만 티어로 토글.

### 16.8 구현 착수 전 "한 줄로 답해야 할" 결정 목록

구현 세션은 코드 짜기 전에 아래를 명시적으로 정하고 시작하라. 안 정하면 도중에 임의 결정이 박힌다.

1. 의미적 충돌(§16.1)은 경고만인가 차단인가 → (권장: 경고만, 차단은 결정론만)
2. 드리프트 시 기본 동작(§16.2)은 덮어쓰기/머지/빌드실패 중 무엇인가 → (권장: 빌드실패)
3. 결정표 검사(§16.3)는 csv 룩업인가 정책엔진인가 → (권장: csv부터)
4. 스키마 확장 경계(§16.4)는 어디까지인가 → (권장: 얇게, 복잡 규칙은 정책 객체로)
5. LLM 출력 검증(§16.5)을 스키마로 강제하는가 → (권장: 예, Pydantic 필수)
6. 규모 티어별 토글(§16.7) 대상은 무엇인가 → (권장: A·B·§9는 항상 ON)

## ✅ 다른 세션을 위한 착수 체크리스트

이 문서로 구현을 시작하는 AI 세션은 다음을 순서대로 확정하고 진행하라.

1. §3 디렉토리 트리를 **그대로** 생성했는가 (이름만 도메인에 맞게).
2. 상태 enum·전이 테이블·불변식이 **domain/ 안에만** 있는가 (§2 판별 통과).
3. 모든 요구사항에 **ID**가 있고 코드·테스트에 `# req:` 주석으로 연결됐는가 (§5.2).
4. 변경마다 **종류(추가/수정/폐기/공존)** 를 분류하고 CHANGELOG를 남기는가 (§5.3/5.4).
5. §9의 4종 테스트가 전부 있는가.
6. 경계 가드(§3.4)가 CI에서 실제로 빌드를 실패시키는가.
7. **변경은 스펙 PR로만 들어오는가** — 코드 직접 수정이 경계 가드로 차단되는가 (§10.2).
8. **충돌 시 게이트가 자동으로 머지를 차단**하는가 — 사람이 무시하고 진행 불가능한가 (§10.3).

> 위 8개가 충족되면, 세션이 달라도 산출물이 수렴하고, 규율이 사람의 미덕이 아니라 시스템의 속성이 된다.
