## 1. 프로젝트 개요

### 1.1 한 줄 정의
관심 단지의 **실거래가 추이 + 실시간 금리 + 대출/세금 규제**를 결합해, "이 금리 환경에서 연 몇 % 올라야 본전인지(손익분기 상승률)"와 레버리지 수익률(ROE)을 자동 계산해주는 개인용 의사결정 도구.

### 1.2 핵심 문제의식
- 매매 동기는 **수익성**: "대출 이자를 감안해도 집값 상승분이 더 크다"는 가설 검증이 목표.
- 단, 2026년 현재는 ① 금리 인상 사이클 진입(기준금리 2.5% → 최종 3.5% 전망), ② 10.15 대책으로 대출 한도 차등화 + 스트레스 DSR 3.0%, ③ 토지거래허가구역으로 갭투자 차단 — **레버리지 마진이 얇아지는 국면**이라 정밀 계산이 필수.
- "분석 다 했는데 매물이 없으면 헛수고" → **매물 유무를 분석의 게이트(트리거)로 두는 설계**가 핵심 차별점.

### 1.3 설계 철학
- **매물은 조사의 끝이 아니라 시작 신호** — 매물이 떠야 분석을 돌린다(이벤트 드리븐).
- **호가보다 실거래가** — 호가는 허수가 많음. 합법적·안정적인 공공 실거래가 + 거래량을 1차 근거로.
- 개인용 우선, 추후 오픈소스화 고려.

## 2. 핵심 사용자 시나리오

```
[로그인]
   ↓
[관심 리스트] ── 또는 ── [서울시 구 선택 → 동 선택 → 아파트 리스트]
   ↓
[단지/동 단위 분석 실행]
   ↓  (실거래 추이 · 거래량 · 금리 · 손익분기 상승률 · 레버리지 ROE)
[매물 게이트] 거래량 바닥(매물 프록시)=잠김 경고 → 깊은 분석 보류
   ↓
[임장 결정]
```

데이터 갱신은 사용자가 UI에서 **업데이트 버튼**으로 트리거 → 외부 API 호출 → DB upsert.

## 3. 화면(UI) 구성

> 전 화면 **모바일 우선 반응형** 필수. 출퇴근 중 폰으로 확인하는 게 주 사용 환경.

### 3.1 로그인 페이지
- 이메일/소셜 로그인(추후 OAuth). MVP는 단일 계정 또는 간단 JWT.
- 로그인 후 기본 진입점은 "내 관심 리스트".

### 3.2 관심 리스트 페이지
- 등록한 단지 카드 목록(단지명 · 최근 실거래가 · 전월 대비 변동 · 매물 유무 뱃지).
- 각 카드 탭 → 단지 분석 상세로 이동.
- 매물 "잠김" 단지는 회색 처리 + 뱃지로 구분.

### 3.3 지역 탐색 (구 → 동 → 아파트)
- **구 선택**: 서울시 25개 자치구 그리드/드롭다운.
- **동 선택**: 선택한 구의 법정동/행정동 리스트.
- **아파트 리스트**: 선택한 동의 단지 목록(거래 빈도순 정렬). "관심 추가" 버튼.
- 모바일에서는 단계별 풀스크린 전환, 데스크탑은 3단 컬럼.

### 3.4 분석 상세 페이지 (동 / 아파트 단위)
- **실거래 추이 차트**: 최근 N개월 평균 거래가 + 거래량(막대) 오버레이.
- **손익분기 패널**: 자기자본 · 대출액 · 금리 입력 → "연 N% 상승해야 본전" 자동 계산.
- **레버리지 ROE 시뮬레이터**: 가정 상승률 슬라이더 → 세전/세후 수익률.
- **규제 알림**: 해당 단지 토허제 여부, 가격대별 대출 한도, 스트레스 DSR 적용 안내.
- **매물 게이트 배너**: MVP는 **거래량 프록시 기반**(최근 거래량 바닥=잠김 경고). 게이트 통과 전엔 깊은 분석 보류. 매물 카운트(비공식 데이터)는 3차에서 보조 강화.

### 3.5 데이터 업데이트 컨트롤 (UI 버튼)
- "실거래가 갱신" / "금리 갱신" / "아파트 리스트 갱신" 버튼.
- 클릭 시 백엔드 수집 작업 트리거 → 진행 상태 표시 → 완료 시 DB upsert 결과(추가/갱신 건수) 토스트.
- 자동 배치(CronJob)와 병행: 평상시 자동, 필요시 수동 강제 갱신.

## 4. 외부 데이터 / API

| 구분 | 소스 | 제공 형태 | 용도 | 비고 |
|------|------|-----------|------|------|
| 매매 실거래가 | 국토교통부 (공공데이터포털 data.go.kr) | REST, XML/JSON, 무료 | 단지별 시세 추이·거래량 | **핵심 소스** |
| 매매가격지수/변동률 | 한국부동산원 R-ONE | 통계 API | 거시 사이클 분석 | 지역별 지수 시계열 |
| 금리(기준/주담대/국고채) | 한국은행 ECOS | REST, 무료 키 | 손익분기 계산의 금리 축 | 실효금리 추정 |
| 행정구역(구/동) | 공공데이터포털 / 행안부 | 코드 데이터 | 구→동→단지 드릴다운 | 초기 1회 적재 |
| 좌표·역세권 거리 | 카카오/네이버 지도 API | REST | 입지(역세권) 보조 지표 | 선택 |
| 교통 호재(GTX 등) | 정형 API 없음 | 수동 데이터셋 | 알파 요인 | 수기 관리 |

### 4.1 현재 매물(호가) 데이터의 한계 — 중요
- **공식 오픈 API 없음.** 네이버 부동산이 사실상 표준이나 공개 API 미제공.
- 비공식 내부 JSON 엔드포인트는 약관 회색지대 + 차단 리스크 → **개인용 한정**, 공개 서비스화 비권장.
- **현실 전략**:
  1. 매물 "유무/카운트"만 가볍게 감지(0→1 변화 알림). 전체 데이터 저장·재배포 X.
  2. **거래량을 매물 프록시로** 활용 — 실거래 건수 바닥 = "매물 잠김" 사전 필터.
  3. 최종 호가 확인은 사용자가 네이버에서 수동(자동화 가치 낮음).

## 5. 분석 로직 (핵심 계산)

### 5.1 손익분기 상승률
```
손익분기 상승률(%) = 대출비중 × 실효금리
예) 대출 6억/매매 10억(60%) × 금리 5% = 연 3.0% 이상 올라야 본전
```

### 5.2 레버리지 ROE
```
순익 = (매매가 × 가정상승률) − (대출액 × 실효금리)
ROE  = 순익 / 자기자본
예) 10억, 자기자본 4억, 대출 6억@5%, 상승 5%:
    순익 = 5,000만 − 3,000만 = 2,000만 → ROE 5.0%
```

### 5.3 규제 룰 엔진 (2026 기준, 변동 가능)
- **대출 한도 차등**: 시가 15억 초과 → 최대 4억, 25억 초과 → 최대 2억(그 외 6억 기준).
- **스트레스 DSR 3.0%**: 실제 금리 + 3.0%p로 한도 심사 → 대출 가능액 축소 반영.
- **토지거래허가구역**: 해당 시 실거주 의무 → 갭투자 시나리오 비활성화.
- **양도세**: 보유/거주 요건(1주택 비과세 2년) 충족 여부로 세후 수익 보정.

> ⚠️ 세제·대출 규제는 자주 바뀜. 룰은 설정값(config)으로 분리해 갱신 쉽게 관리. 실행 전 세무사·은행 확인 안내 문구 노출.
>
> 📐 구현(HexaArch): 위 룰은 `regulation/decision_tables/*.csv` **결정표**(§3.5)로 표현 — 대출한도는 가격 **구간(range)**(15억/25억 경계), DSR·토허제는 플래그. 규제 개정 대비 `effective_date`/버전 컬럼으로 **구버전·신버전 공존**(§5.3 조건부 공존 / §16.6). 결정표 빈칸·구간 겹침/공백은 `check`가 차단.

## 6. 시스템 아키텍처

### 6.1 구성요소
```
[React PWA (반응형)]  ──REST──>  [DRF API 서버]  ──>  [PostgreSQL]
                                      │
                                      ├─ 수집 워커(배치/수동 트리거)
                                      │     ├─ 국토부 실거래가
                                      │     ├─ ECOS 금리
                                      │     └─ 매물 카운트 폴링(개인용)
                                      └─ 규제 룰 엔진 / 분석 계산
```

### 6.2 백엔드 (DRF)
- **수집 잡**: 국토부·ECOS 주기 수집 → 정규화 → `upsert`(단지·실거래·금리 테이블).
- **분석 엔드포인트**: 단지 입력 → 손익분기·ROE·DSR 한도·규제 플래그 반환.
- **트리거 엔드포인트**: UI 버튼 → 수집 잡 비동기 실행(상태 폴링/웹소켓).
- ViewSet 기반 리소스 분리(관심리스트/지역/단지/실거래/금리/분석).
- ⚠️ **헥사고날 경계(HexaArch §2)**: 손익분기·ROE·DSR 한도·규제 판정 등 *모든 계산/판단은 도메인 코어*(`valuation`/`regulation`)에 둔다. DRF view/serializer는 유스케이스 호출·입출력 변환만(판단 금지). 외부 API 호출·DB upsert는 아웃바운드 포트 뒤로.

### 6.3 데이터 모델 (초안)
- `Region(gu, dong)` · `Complex(단지, region_fk, 좌표)` · `Trade(complex_fk, 면적, 금액, 층, 계약일)`
- `Rate(종류, 값, 기준일)` · `Watchlist(user_fk, complex_fk)` · `ListingSnapshot(complex_fk, count, 수집시각)`
- 실거래·금리는 (자연키, 기준일) 유니크 제약으로 idempotent upsert.

### 6.4 인프라 (현재 목표: git 관리 + 로컬 개발)
- **현재 내부 배포 목적 없음.** 프로젝트는 git에 올려 관리하고, 개발/실행은 `docker compose` 로컬로 한다.
- 배포(K3s + ArgoCD + 사설레지스트리 + Cloudflare)는 **추후 옵션** — 필요해지면 기존 서버 스택으로 승격. (EKS 아님. 원안의 EKS는 K3s로 정정.)
- 수집 배치는 (배포 시) **CronJob**, API는 Deployment로 매핑 예정.
- 수동 트리거 잡은 **Celery + Redis**로 비동기 처리(상태 폴링). → `docker compose`에 Redis 추가.

### 6.5 반응형/PWA
- 모바일 우선 레이아웃, 브레이크포인트로 데스크탑 3단 확장.
- PWA(홈 화면 추가, 매물 0→1 알림은 추후 푸시 연동 여지).

### 6.6 핵심 상태 모델 (헥사고날 코어 — rev1 신규 명시)

> 이 플랫폼은 본질이 수집·계산이라 상태기계가 약해 보이지만, 판단/무결성이 집중되는 상태기계가 **두 개** 있다.
> 둘 다 도메인 코어(`domain/`)에 박는다 — 방어선 A·B의 실제 거처.

- **수집 잡 `CollectionJob`** (context: `collection`)
  - 상태: `IDLE → RUNNING → SUCCEEDED / FAILED`
  - 전이 규칙: RUNNING 중 재트리거·동시 실행 거부(업데이트 버튼 연타 방어).
  - 불변식: 실거래·금리는 (자연키 + 기준일) 유니크 → 멱등 upsert. 깨진 부분수집 결과로는 SUCCEEDED 불가.
- **분석 게이트 `AnalysisEligibility`** (context: `watchlist`/`valuation` 경계)
  - 상태: `LOCKED(거래량 바닥) → READY → ANALYZED`
  - 불변식: LOCKED에선 깊은 분석 산출물 생성 금지(게이트 우회 차단).
  - 불변식: 손익분기/ROE는 `대출액 ≤ 규제 한도`일 때만 유효(규제 위반 시나리오로는 산출 불가).
  - 불변식: 토허제 단지는 갭투자 시나리오 비활성(상태 의존).

## 7. MVP 범위

### 7.1 1차(반드시)
1. 로그인(간단 JWT) + 관심 리스트.
2. 구 → 동 → 아파트 드릴다운.
3. 단지 분석: 국토부 실거래 추이 + ECOS 금리 → **손익분기 상승률 자동 계산**.
4. "실거래가 갱신" / "금리 갱신" 버튼 → DB upsert.
5. 반응형(폰 우선).

### 7.2 2차
- 레버리지 ROE 시뮬레이터, 규제 룰 엔진(DSR·한도·토허제).
- 거래량 기반 "매물 잠김" 지표.

### 7.3 3차
- 매물 카운트 폴링 + 0→1 알림(개인용), 지도/역세권, 교통 호재 데이터셋.
- PWA 푸시 알림.

## 8. 리스크 & 유의사항
- **매물 호가 데이터 비공식성** → 개인용 한정, 서비스화 시 법적·차단 리스크. 거래량 프록시로 대체 설계.
- **규제·세제 변동성** → 룰을 코드가 아닌 설정으로 분리, 면책 문구 노출.
- **금리 전망 가변성** → 분기마다 갱신, 실행 시점 은행 금리표 재확인 안내.
- **API 호출 제한** → 수집 배치 주기·캐싱으로 쿼터 관리.
